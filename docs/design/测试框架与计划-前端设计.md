| **@vitest/coverage-v8**      | 代码覆盖率           | ^3.2.x  |

### 为什么选 Vitest 而非 Jest？

- 与 Vite 共享配置（路径别名、SWC 编译），零额外配置
- 原生 ESM 支持，与项目 `"type": "module"` 一致
- HMR 热更新模式，开发体验更快
- API 与 Jest 兼容，迁移成本低

## 三、目录结构与约定

### 3.1 测试文件位置

采用 **`__tests__/` 子目录** 策略，测试文件放在与源文件同目录下的 `__tests__/` 子文件夹，**不与源文件同级放置**：

```
src/
├── utils/
│   ├── format-number.ts
│   ├── format-time.ts
│   └── __tests__/                    ← 测试子目录
│       ├── format-number.test.ts
│       └── format-time.test.ts
├── api/
│   ├── client.ts
│   └── __tests__/
│       └── client.test.ts
├── auth/
│   ├── auth-reducer.ts
│   └── __tests__/
│       └── auth-reducer.test.ts
├── permission/
│   └── __tests__/
│       └── use-permission.test.ts
└── test/                             ← 测试基础设施（setup、工具函数）
    ├── setup.ts                      ← Vitest 全局 setup
    └── test-utils.tsx                ← 自定义 render 和工具
```

测试文件引用源文件时使用相对路径 `../`：

```ts
// ✅ 正确（从 __tests__/ 向上一层）
import { fNumber } from '../format-number';
import { tokenStorage } from '../client';

// ❌ 错误
import { fNumber } from './format-number';
```

### 3.2 命名约定

| 类型       | 文件名格式                        | 示例                        |
| ---------- | --------------------------------- | --------------------------- |
| 单元测试   | `<模块名>.test.ts`                | `format-number.test.ts`     |
| 组件测试   | `<组件名>.test.tsx`               | `label.test.tsx`            |
| 集成测试   | `<功能名>.integration.test.tsx`   | `auth-flow.integration.test.tsx` |

### 3.3 脚本命令

```bash
npm test              # 运行全部测试（CI 使用）
npm run test:watch    # 开发时 watch 模式
npm run test:coverage # 生成覆盖率报告
npm run test:ui       # Vitest 可视化 UI
```

## 五、Mock 策略

### 5.1 API Mock（MSW）

使用 MSW (Mock Service Worker) 拦截网络请求，在测试环境中模拟后端 API：

```typescript
// src/test/mocks/handlers.ts
import { http, HttpResponse } from 'msw';

export const handlers = [
  http.post('/api/auth/login', () =>
    HttpResponse.json({ code: 0, data: { accessToken: 'mock-token' } })
  ),
  http.post('/api/auth/refresh', () =>
    HttpResponse.json({ code: 0, data: { accessToken: 'refreshed-token' } })
  ),
];
```

### 5.2 浏览器 API Mock

| API              | Mock 方式                              |
| ---------------- | -------------------------------------- |
| `fetch`          | MSW 拦截，或 `vi.fn()` 手动 mock       |

| `localStorage`   | jsdom 自带                             |

## 八、最佳实践

1. **测试文件放在 `__tests__/` 子目录** — 不与源文件同级，目录结构清晰，import 用 `../`
2. **每个 `describe` 块对应一个函数或组件** — 结构清晰
3. **使用 `test-utils.tsx` 中的 `renderWithProviders`** — 统一 Provider 包裹
4. **API 测试使用 MSW** — 接近真实网络行为，避免 mock 泄漏
5. **避免测试实现细节** — 测试行为和输出，而非内部状态
6. **保持测试独立** — 每个 test case 不依赖其他 test 的执行顺序
7. **null/undefined/边界值** — 每个函数至少测试边界情况
8. **修改源文件必须同步更新测试** — 新增/修改/删除函数都需同步维护 `__tests__/` 中对应用例
