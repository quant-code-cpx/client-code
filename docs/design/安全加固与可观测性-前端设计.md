## 一、设计目标

将 `docs/前端待办清单.md` 中「安全加固」和「监控与错误追踪」两大类共 10 项待办落地为可执行的前端实现方案，供 AI 代码生成模型直接参考并生成代码。

**核心原则**：

1. **最小侵入** — 不修改已有组件的对外 API，只在内部加安全层
2. **零运行时依赖膨胀** — 仅新增 `dompurify`（~7 kB gzip）和 `@sentry/react`（tree-shakable）
3. **纵深防御** — 同一风险点在多个层面设置防线（如 XSS：后端校验 + 前端 sanitize + CSP）
4. **可渐进实施** — 每个小节独立可交付，互不阻塞

## 三、XSS 防护：dangerouslySetInnerHTML 安全化

### 3.1 问题定位

**文件**：`src/sections/auth/sign-in-view.tsx`（第 148–158 行）

```tsx
<Box
  component="span"
  dangerouslySetInnerHTML={{
    __html: captcha.svgImage
      .replace(/(<svg\b[^>]*?)\s+width="[^"]*"/i, '$1')
      // ... 其他 replace
  }}
/>
```

后端返回的 `svgImage` 字符串直接注入 DOM，若后端被攻陷或响应被篡改，可执行任意 JS。

### 3.2 解决方案

**新增依赖**：`dompurify`（`^3.2.4`）+ `@types/dompurify`（dev）

**新建工具函数**：`src/utils/sanitize.ts`

```ts
import DOMPurify from 'dompurify';

// ------------------------------------------------------------------
// HTML / SVG 净化工具
// ------------------------------------------------------------------

/**
 * 净化 SVG 字符串，仅保留安全的 SVG 标签和属性。
 * 用于后端返回的 SVG 内容在前端通过 dangerouslySetInnerHTML 渲染的场景。
 */
export function sanitizeSvg(dirty: string): string {
  return DOMPurify.sanitize(dirty, {
    USE_PROFILES: { svg: true, svgFilters: true },
    ADD_TAGS: ['svg', 'path', 'rect', 'circle', 'ellipse', 'line', 'polyline', 'polygon', 'text', 'tspan', 'g', 'defs', 'use'],
    ADD_ATTR: ['viewBox', 'preserveAspectRatio', 'xmlns', 'fill', 'stroke', 'd', 'cx', 'cy', 'r', 'rx', 'ry', 'x', 'y', 'x1', 'y1', 'x2', 'y2', 'width', 'height', 'transform', 'opacity', 'font-size', 'font-family', 'text-anchor', 'style'],
    FORBID_TAGS: ['script', 'iframe', 'object', 'embed', 'form', 'input'],
    FORBID_ATTR: ['onerror', 'onload', 'onclick', 'onmouseover'],
  });
}

/**
 * 净化普通 HTML 字符串（如富文本内容）。
 * 保留基本排版标签，移除所有脚本和事件处理器。
 */
export function sanitizeHtml(dirty: string): string {
  return DOMPurify.sanitize(dirty, {
    USE_PROFILES: { html: true },
    FORBID_TAGS: ['script', 'iframe', 'object', 'embed', 'form'],
    FORBID_ATTR: ['onerror', 'onload', 'onclick', 'onmouseover', 'onfocus', 'onblur'],
  });
}
```

### 3.3 改造 sign-in-view.tsx

```tsx
// 改造前：
dangerouslySetInnerHTML={{ __html: captcha.svgImage.replace(...) }}

// 改造后：
import { sanitizeSvg } from 'src/utils/sanitize';

const safeSvg = sanitizeSvg(
  captcha.svgImage
    .replace(/(<svg\b[^>]*?)\s+width="[^"]*"/i, '$1')
    .replace(/(<svg\b[^>]*?)\s+height="[^"]*"/i, '$1 width="100%" height="100%"')
    .replace(/(<svg\b[^>]*?)\s+preserveAspectRatio="[^"]*"/i, '$1')
    .replace(/<svg\b/i, '<svg preserveAspectRatio="xMidYMid slice"')
);

<Box component="span" dangerouslySetInnerHTML={{ __html: safeSvg }} />
```

### 3.4 ApexCharts innerHTML（无需修改）

`src/components/chart/chart.tsx` 中的 `containerEl.innerHTML = ''` 仅用于清空容器，不渲染用户内容，**无安全风险**。

## 四、输入内容 Sanitize

### 4.1 设计思路

分两层防护：

1. **显示层** — 使用 DOMPurify 对所有用户输入的展示进行转义
2. **传输层** — 在 API 请求拦截器中对字符串类型字段做基本 XSS 字符转义

### 4.2 新建工具函数：`src/utils/sanitize.ts`（追加）

```ts
/**
 * 转义 HTML 特殊字符，用于将用户输入安全地嵌入 DOM。
 * 适用于不需要 DOMPurify 的轻量场景（如 <input> 值回显到文本节点）。
 */
export function escapeHtml(str: string): string {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#039;');
}

/**
 * 递归清理对象中所有字符串值的 XSS 危险字符。
 * 用于 API 请求 body 的预处理。
 */
export function sanitizeRequestBody<T>(body: T): T {
  if (body === null || body === undefined) return body;
  if (typeof body === 'string') return escapeHtml(body) as T;
  if (Array.isArray(body)) return body.map(sanitizeRequestBody) as T;
  if (typeof body === 'object') {
    const result: Record<string, unknown> = {};
    for (const [key, value] of Object.entries(body as Record<string, unknown>)) {
      result[key] = sanitizeRequestBody(value);
    }
    return result as T;
  }
  return body;
}
```

### 4.3 可选：API Client 拦截器集成

> **注意**：是否在 `apiClient` 层自动 sanitize 取决于后端是否已做输入校验。
> 若后端已使用 `class-validator` + `class-transformer` 管道校验，此步骤可设置为仅开发环境警告模式。

在 `src/api/client.ts` 的 `request()` 函数中增加可选预处理：

```ts

const serializedBody = safeBody !== undefined ? JSON.stringify(safeBody) : undefined;
```

**开关控制**（通过环境变量）：

```ts

```

## 五、CSP（Content Security Policy）配置

### 5.1 CSP 策略设计

本项目使用 Emotion（CSS-in-JS），需要 `'unsafe-inline'` 或 nonce 方案用于 style。采用 `'unsafe-inline'` 策略（Emotion 暂不支持全面 nonce）。

### 5.2 vercel.json 配置

**修改文件**：`vercel.json`

```json
{
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/"
    }
  ],
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        {
          "key": "Content-Security-Policy",
          "value": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; font-src 'self' data:; connect-src 'self' wss: ws:; frame-ancestors 'none'; base-uri 'self'; form-action 'self'; object-src 'none'"
        },
        {
          "key": "X-Content-Type-Options",
          "value": "nosniff"
        },
        {
          "key": "X-Frame-Options",
          "value": "DENY"
        },
        {
          "key": "Referrer-Policy",
          "value": "strict-origin-when-cross-origin"
        },
        {
          "key": "Permissions-Policy",
          "value": "camera=(), microphone=(), geolocation=()"
        }
      ]
    }
  ]
}
```

### 5.3 CSP 指令说明

| 指令 | 值 | 说明 |
|------|----|------|
| `default-src` | `'self'` | 默认只允许同源资源 |
| `script-src` | `'self'` | 只允许同源脚本（Vite bundle） |
| `style-src` | `'self' 'unsafe-inline'` | Emotion CSS-in-JS 需要 inline style |
| `img-src` | `'self' data: blob:` | 允许 data URI（SVG captcha）和 blob（图表导出） |
| `font-src` | `'self' data:` | 本地字体文件（Barlow、DM Sans） |
| `connect-src` | `'self' wss: ws:` | API 请求 + WebSocket 连接 |
| `frame-ancestors` | `'none'` | 禁止被 iframe 嵌套（防 Clickjacking） |
| `object-src` | `'none'` | 禁止 Flash / Java 插件 |

### 5.4 开发环境 CSP（可选）

在 `vite.config.ts` 中添加开发环境 CSP meta tag 用于本地调试：

```ts
// vite.config.ts → plugins 数组中新增
{
  name: 'csp-meta-tag',
  transformIndexHtml(html) {
    if (process.env.NODE_ENV !== 'production') {
      return html.replace(
        '</head>',
        `<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; font-src 'self' data:; connect-src 'self' ws://localhost:*;">\n</head>`
      );
    }
    return html;
  },
}
```

> 开发环境额外允许 `'unsafe-eval'`（Vite HMR 需要）和 `ws://localhost:*`。

## 六、CSRF Token 集成

### 6.1 前提

CSRF 防护需要后端配合。后端如使用 `csurf` 或 NestJS `@nestjs/csrf` 中间件，通常在初次 GET 请求或 Cookie 中下发 CSRF Token。

### 6.2 方案设计

**策略**：采用「Double Submit Cookie」模式 — 后端将 CSRF Token 写入非 HttpOnly Cookie（如 `XSRF-TOKEN`），前端读取后放入请求头。

**修改文件**：`src/api/client.ts`

```ts
// ---------- CSRF Token 辅助 ----------

/**
 * 从 Cookie 中读取 CSRF Token。
 * 后端通过 Set-Cookie: XSRF-TOKEN=<token>; Path=/ 下发（非 HttpOnly）。
 */
function getCsrfToken(): string | null {
  const match = document.cookie.match(/(?:^|;\s*)XSRF-TOKEN=([^;]*)/);
  return match ? decodeURIComponent(match[1]) : null;
}
```

在 `request()` 函数的 headers 构造中添加：

```ts
const csrfToken = getCsrfToken();

const headers: Record<string, string> = {
  'Content-Type': 'application/json',
  ...(options.headers as Record<string, string>),
  ...(token ? { Authorization: `Bearer ${token}` } : {}),
  ...(csrfToken ? { 'X-XSRF-TOKEN': csrfToken } : {}),
};
```

### 6.3 何时启用

- 后端未启用 CSRF 中间件时：`getCsrfToken()` 返回 `null`，header 不附加，**无副作用**
- 后端启用后：自动生效，无需前端额外配置

## 七、敏感字段前端脱敏

### 7.1 脱敏规则

| 字段类型 | 脱敏规则 | 示例 |
|----------|----------|------|
| 手机号 | 中间 4 位替换为 `****` | `138****1234` |
| 邮箱 | `@` 前保留首尾字符 | `u***r@example.com` |
| 身份证号 | 保留前 3 后 4 | `110***********1234` |
| API Key / Token | 仅显示前 4 后 4 | `sk-a****...b8c9` |

### 7.2 新建工具函数：`src/utils/mask.ts`

```ts
// ------------------------------------------------------------------
// 敏感字段脱敏工具
// ------------------------------------------------------------------

/** 手机号脱敏：138****1234 */
export function maskPhone(phone: string): string {
  if (phone.length < 7) return phone;
  return `${phone.slice(0, 3)}****${phone.slice(-4)}`;
}

/** 邮箱脱敏：u***r@example.com */
export function maskEmail(email: string): string {
  const [local, domain] = email.split('@');
  if (!domain || local.length <= 2) return email;
  return `${local[0]}${'*'.repeat(Math.min(local.length - 2, 3))}${local[local.length - 1]}@${domain}`;
}

/** 通用密钥/Token脱敏：sk-a****...b8c9 */
export function maskSecret(secret: string, prefixLen = 4, suffixLen = 4): string {
  if (secret.length <= prefixLen + suffixLen) return '*'.repeat(secret.length);
  return `${secret.slice(0, prefixLen)}****...${secret.slice(-suffixLen)}`;
}
```

### 7.3 使用场景

- 用户管理列表中的邮箱、手机号
- 系统设置页面的 API Key 展示
- 日志中避免打印完整 Token

## 八、存储安全审计

### 8.1 审计结果

| 存储位置 | Key | 内容 | 是否敏感 | 处置 |
|----------|-----|------|----------|------|
| localStorage | `quant-client-theme-preset` | 主题名称字符串 | ❌ | 保持不变 |
| 内存变量 | `_accessToken` | JWT Access Token | ✅ | ✅ 已采用内存存储 |

### 8.2 防护措施确认

1. **Access Token 不进 localStorage** — 已实现（`src/api/client.ts` 第 8–22 行）
2. **Refresh Token 为 HttpOnly Cookie** — 后端负责设置
3. **BroadcastChannel 同源限制** — 浏览器原生保障，无跨域泄露风险
4. **无敏感数据存入 sessionStorage** — 已确认无使用

### 8.3 新增防护：存储安全检查工具（可选）

> 作为开发环境辅助工具，在控制台检查是否有敏感数据意外存入 Storage。

```ts
// src/utils/storage-audit.ts（仅开发环境加载）

export function auditLocalStorage(): void {
  if (import.meta.env.PROD) return;

  const sensitivePatterns = [
    /token/i, /password/i, /secret/i, /key/i,
    /auth/i, /session/i, /credential/i,
  ];

  const allowedKeys = new Set(['quant-client-theme-preset']);

  for (let i = 0; i < localStorage.length; i++) {
    const key = localStorage.key(i);
    if (!key || allowedKeys.has(key)) continue;

    const isSuspicious = sensitivePatterns.some((p) => p.test(key));
    if (isSuspicious) {
      console.warn(
        `[Storage Audit] localStorage 中发现可疑 key: "${key}"，请确认是否包含敏感数据。`
      );
    }
  }
}
```

在 `src/main.tsx` 中开发环境调用：

```ts
if (import.meta.env.DEV) {
  import('src/utils/storage-audit').then(({ auditLocalStorage }) => auditLocalStorage());
}
```

## 九、WebSocket 鉴权加固

### 9.1 问题定位

**文件**：`src/lib/socket.ts`

当前 Socket.IO 客户端未传递 auth token，仅依赖 HttpOnly Cookie。虽然 Cookie 在同源下会自动携带，但：

- WebSocket 升级请求的 Cookie 行为因浏览器和代理不同可能不一致
- 显式传递 token 提供更可靠的鉴权保障

### 9.2 改造方案

**修改文件**：`src/lib/socket.ts`

```ts
import { io, type Socket } from 'socket.io-client';

import { tokenStorage } from 'src/api/client';

// ------------------------------------------------------------------
// Socket.io 客户端单例
// ------------------------------------------------------------------

const WS_BASE_URL = (import.meta.env.VITE_WS_URL as string | undefined) ?? '';

let _socket: Socket | null = null;

export function getSocket(): Socket {
  if (!_socket) {
    _socket = io(`${WS_BASE_URL}/ws`, {
      autoConnect: false,
      transports: ['websocket', 'polling'],
      path: '/socket.io',
      // 鉴权：每次连接/重连时动态获取最新 token
      auth: (cb) => {
        cb({ token: tokenStorage.get() });
      },
    });

    // Token 刷新后更新 socket auth（下次重连时自动使用新 token）
    _socket.on('connect_error', (err) => {
      if (err.message === 'unauthorized' || err.message === 'jwt expired') {
        // Token 已过期，断开等待 API 层刷新后由页面逻辑重新 connect
        _socket?.disconnect();
      }
    });
  }
  return _socket;
}

export function destroySocket(): void {
  if (_socket) {
    _socket.disconnect();
    _socket = null;
  }
}
```

### 9.3 后端配合

后端 Socket.IO Gateway 需在 `handleConnection` 中验证 `client.handshake.auth.token`：

```ts
// server-side 示意（NestJS）
@WebSocketGateway({ namespace: '/ws' })
export class EventsGateway {
  handleConnection(client: Socket) {
    const token = client.handshake.auth?.token;
    if (!token || !this.jwtService.verify(token)) {
      client.emit('error', 'unauthorized');
      client.disconnect();
    }
  }
}
```

## 十、前端错误追踪（Sentry 集成）

### 10.1 依赖

**新增依赖**：`@sentry/react`（`^9.x`）

> `@sentry/react` 内置 React 错误边界、性能追踪和 Session Replay 功能。v9+ 支持 tree-shaking，实际 bundle 增量约 30 kB gzip。

### 10.2 环境变量

```env
# .env.production
VITE_SENTRY_DSN=https://<key>@<org>.ingest.sentry.io/<project-id>
VITE_SENTRY_ENVIRONMENT=production

# .env.development（可选，开发环境默认不上报）
VITE_SENTRY_DSN=
```

### 10.3 初始化：`src/lib/sentry.ts`

```ts
import * as Sentry from '@sentry/react';

import { CONFIG } from 'src/config-global';

// ------------------------------------------------------------------
// Sentry 初始化
// 仅在配置了 DSN 时启用（生产环境），开发环境跳过
// ------------------------------------------------------------------

const SENTRY_DSN = import.meta.env.VITE_SENTRY_DSN as string | undefined;

export function initSentry(): void {
  if (!SENTRY_DSN) return;

  Sentry.init({
    dsn: SENTRY_DSN,
    environment: (import.meta.env.VITE_SENTRY_ENVIRONMENT as string) || 'development',
    release: `quant-client@${CONFIG.appVersion}`,

    // 性能监控采样率（生产环境建议 0.1-0.3）
    tracesSampleRate: 0.2,

    // Session Replay 采样率
    replaysSessionSampleRate: 0,
    replaysOnErrorSampleRate: 1.0,

    integrations: [
      Sentry.browserTracingIntegration(),
      Sentry.replayIntegration({ maskAllText: false, blockAllMedia: false }),
    ],

    // 过滤非关键错误
    beforeSend(event) {
      // 忽略 ResizeObserver 错误（浏览器 bug，非应用问题）
      if (event.exception?.values?.[0]?.value?.includes('ResizeObserver')) {
        return null;
      }
      return event;
    },
  });
}

/** 设置当前用户信息，用于 Sentry 事件关联 */
export function setSentryUser(user: { id: string; username?: string; role?: string }): void {
  if (!SENTRY_DSN) return;
  Sentry.setUser({ id: user.id, username: user.username, segment: user.role });
}

/** 清除用户信息（登出时调用） */
export function clearSentryUser(): void {
  if (!SENTRY_DSN) return;
  Sentry.setUser(null);
}
```

## 十一、Web Vitals 监控

### 11.1 方案

使用浏览器原生 `web-vitals` API（或 Sentry 内置的 Web Vitals 追踪）采集核心指标。

> `@sentry/react` v9 的 `browserTracingIntegration()` 已内置 Web Vitals 采集（LCP、FID/INP、CLS），无需额外依赖。

### 11.2 自定义上报（可选独立方案）

若不使用 Sentry，可独立采集并上报：

**新建文件**：`src/lib/web-vitals.ts`

```ts
// ------------------------------------------------------------------
// Web Vitals 采集与上报
// 使用 Performance Observer API，零依赖
// ------------------------------------------------------------------

type VitalMetric = {
  name: string;
  value: number;
  rating: 'good' | 'needs-improvement' | 'poor';
};

type ReportCallback = (metric: VitalMetric) => void;

let reportFn: ReportCallback | null = null;

export function initWebVitals(callback: ReportCallback): void {
  reportFn = callback;

  // LCP (Largest Contentful Paint)
  observePerformanceEntry('largest-contentful-paint', (entries) => {
    const last = entries[entries.length - 1] as PerformanceEntry & { startTime: number };
    reportFn?.({
      name: 'LCP',
      value: last.startTime,
      rating: last.startTime < 2500 ? 'good' : last.startTime < 4000 ? 'needs-improvement' : 'poor',
    });
  });

  // CLS (Cumulative Layout Shift)
  let clsValue = 0;
  observePerformanceEntry('layout-shift', (entries) => {
    for (const entry of entries) {
      const shift = entry as PerformanceEntry & { hadRecentInput: boolean; value: number };
      if (!shift.hadRecentInput) clsValue += shift.value;
    }
    reportFn?.({
      name: 'CLS',
      value: clsValue,
      rating: clsValue < 0.1 ? 'good' : clsValue < 0.25 ? 'needs-improvement' : 'poor',
    });
  });

  // FID (First Input Delay) / INP
  observePerformanceEntry('first-input', (entries) => {
    const entry = entries[0] as PerformanceEntry & { processingStart: number; startTime: number };
    const fid = entry.processingStart - entry.startTime;
    reportFn?.({
      name: 'FID',
      value: fid,
      rating: fid < 100 ? 'good' : fid < 300 ? 'needs-improvement' : 'poor',
    });
  });
}

function observePerformanceEntry(
  type: string,
  callback: (entries: PerformanceEntry[]) => void
): void {
  if (typeof PerformanceObserver === 'undefined') return;
  try {
    const observer = new PerformanceObserver((list) => callback(list.getEntries()));
    observer.observe({ type, buffered: true });
  } catch {
    // 部分浏览器不支持某些 entry type，静默忽略
  }
}
```

### 11.3 集成方式

在 `src/main.tsx` 中初始化：

```ts
import { initWebVitals } from './lib/web-vitals';

// Sentry 方案：Web Vitals 由 browserTracingIntegration 自动采集，无需额外调用

  // 上报到自建后端
  navigator.sendBeacon('/api/metrics/web-vitals', JSON.stringify(metric));
});
```

## 十二、未捕获异常统一上报

### 12.1 全局错误捕获

**新建文件**：`src/lib/global-error-handler.ts`

```ts
import * as Sentry from '@sentry/react';

// ------------------------------------------------------------------
// 全局未捕获错误处理器
// 捕获 window.onerror 和 unhandledrejection，统一上报至 Sentry
// ------------------------------------------------------------------

export function installGlobalErrorHandlers(): void {
  // 未捕获的同步错误
  window.addEventListener('error', (event) => {
    // 忽略跨域脚本错误（无法获取详情）
    if (event.message === 'Script error.') return;

    console.error('[Global Error]', event.error || event.message);
    Sentry.captureException(event.error || new Error(event.message), {
      tags: { handler: 'window.onerror' },
    });
  });

  // 未捕获的 Promise 拒绝
  window.addEventListener('unhandledrejection', (event) => {
    const reason = event.reason;
    console.error('[Unhandled Rejection]', reason);

    if (reason instanceof Error) {
      Sentry.captureException(reason, {
        tags: { handler: 'unhandledrejection' },
      });
    } else {
      Sentry.captureMessage(`Unhandled Promise Rejection: ${String(reason)}`, {
        level: 'error',
        tags: { handler: 'unhandledrejection' },
      });
    }
  });
}
```

### 12.2 集成

在 `src/main.tsx` 中，Sentry 初始化之后：

```ts
import { initSentry } from './lib/sentry';
import { installGlobalErrorHandlers } from './lib/global-error-handler';

initSentry();
installGlobalErrorHandlers();
```

### 12.3 无 Sentry 降级方案

若项目暂不引入 Sentry，将 `Sentry.captureException` 替换为自建上报函数：

```ts
function reportError(error: unknown, context?: Record<string, string>): void {
  // 发送到自建后端的错误收集端点
  navigator.sendBeacon(
    '/api/metrics/client-error',
    JSON.stringify({
      message: error instanceof Error ? error.message : String(error),
      stack: error instanceof Error ? error.stack : undefined,
      context,
      url: window.location.href,
      timestamp: Date.now(),
      userAgent: navigator.userAgent,
    })
  );
}
```
